5. Cost-tier rerank trigger / "~40%" not built as described

"What routes a query to cheap vs expensive rerank? Is the ~40% measured?"

Premise correction (say this): "There's no cheap-vs-expensive rerank tiering — I don't want to invent one. Reranking is always Cohere rerank-v3.5, with a local ms-marco cross-encoder as a fallback only on Cohere error (graceful degradation), not a cost decision. So there's no '40%' rerank figure.

What is gated for cost is the conflict detector: its claim-extraction Sonnet call costs ~14s, so it only fires when the query shows comparison/guidance intent (a keyword gate — 'guidance', 'versus', 'compare', 'revised'…). Plain queries skip it entirely (0 ms). That's the real cost-gating story."

It also unlocks the cross-source conflict detector — transcript claim vs filed number — which is the stronger version of the differentiator."

7. Checkpointer / persistence none

"Are you using a LangGraph checkpointer? Which persistence?"

Honest answer: "No checkpointer. The graph is stateless per request — each query is a fresh StateGraph.invoke(), no MemorySaver/SqliteSaver, no cross-turn persistence. It's a single-shot Q&A pipeline, not a multi-turn agent, so there's no conversation state to checkpoint."

When I'd add one: "A checkpointer earns its place when you have (a) multi-turn conversation memory, (b) long-running/interruptible runs you want to resume, or (c) human-in-the-loop approval mid-graph. None apply to a stateless single-query RAG call. If I added last-N-turn follow-ups, I'd wire MemorySaver (or SqliteSaver for durability) keyed by a session id."

8. Judge model Claude, not GPT-4o

"What's your eval judge model? (your notes said GPT-4o)"

Correction — keep this consistent: "The judge is Claude (sonnet-4-6), not GPT-4o. Two places use an LLM judge: RAGAS faithfulness (Claude judge + Voyage embeddings) and the ablation's relevance-pooling judge (also Claude). GPT-4o appears in my spec as a planned Claude-vs-GPT comparison, but I deliberately skipped that bake-off — GPT-4o is two generations stale now, and a rigorous cross-model comparison needs a bigger labeled set than 50 queries to be meaningful. So: judge = Claude, everywhere, consistently."

Faithfulness result: RAGAS 0.806 (n=40) — passes the 0.80 gate.

9. Conflict detection — dedicated step or prompt instruction? dedicated step

"Is conflict detection a real step, or just an instruction in the synthesis prompt?"

"It's a dedicated node with its own Sonnet call — not a line in the generation prompt. Two steps: (1) a structured extraction call pulls every numeric claim from the evidence (metric, subject, value, period, is_guidance); (2) pure Python pairwise-compares them with per-metric thresholds. Keeping it separate from generation is deliberate — the extraction is structured/verifiable, and I can unit-test the comparison logic without an LLM."

The precision story (your best debugging narrative): naive version fired 6 conflicts on query one — all false positives. Four gates, each killing a class: different-chunk (not guidance ranges like "down 15% to 18%"), comparable-period (not quarter-vs-full-year), same-subject (not Products-8% vs Services-17%), different-date (a real conflict is guidance in one call vs actual in a later one). Result: 6 → 0 false positives; fires correctly on a genuine guided-15%/delivered-9% case.

Thresholds: revenue >3%, EPS >5%, margin >1pp, growth/guidance >2pp. hand-set — reasoned defaults, not calibrated on labeled pairs (the spec's 20+20 calibration is future work; say so if asked).
